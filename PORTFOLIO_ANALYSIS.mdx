# Portfolio Analysis — Sajjad Karim Developer Portfolio

**Document Version:** 1.0  
**Analysis Date:** June 14, 2026  
**Analyst Role:** Principal Staff Frontend Engineer / UI-UX Architect  
**Scope:** Full codebase audit — documentation only, no implementation  
**Target Vision:** World-class premium portfolio ($20,000+ product quality) comparable to Awwwards, Lusion, Linear, Stripe, Vercel, Framer, Apple

---

## Executive Summary

This portfolio is a **Next.js 14 App Router** single-page developer portfolio built primarily in **JavaScript** (not TypeScript). It follows the structure and visual language of a widely circulated open-source "Developer Portfolio" template (originally by Asad Ali Asim), customized with Sajjad Karim's personal data, projects, and branding touches.

**Current state in one sentence:** A functional, dark-themed MERN developer portfolio with code-editor aesthetics, sticky project cards, Lottie animations, and a skills marquee — but it reads as a **customized template** rather than a bespoke premium product.

**Overall assessment:** The foundation is solid for iteration. Content sections exist (Hero, About, Experience, Skills, Projects, Education, Contact), routing is minimal, and the dark violet/pink/cyan palette is cohesive. However, significant gaps prevent award-tier positioning: broken mobile navigation, dead/unused code paths, template attribution in footer, hardcoded third-party credentials, no SEO beyond basic title/description, no motion system, no design tokens, limited project presentation (code-block cards without screenshots), and build failures in the current environment.

**Recommended design direction:** **Vercel × Linear × Stripe** — refined dark minimalism with surgical motion, typographic hierarchy, and case-study-grade project storytelling. Avoid over-indexing on the current "terminal/code editor" gimmick; evolve it into a subtle signature rather than the dominant motif.

## Installed Packages

### Production Dependencies

#### `@emailjs/browser` (^4.4.1)
- **Purpose:** Send contact form emails from the browser without a backend.
- **Used:** Yes — `contact-form.jsx` calls `emailjs.send()` on submit.
- **Should remove:** No — but **should migrate** to server-side `/api/contact` for security.
- **Should upgrade:** Keep current; or remove once server route is wired.
- **Critical issue:** Hardcoded service ID, template ID, public key, and recipient email (`asadaliasim909@gmail.com`) — template author credentials, not Sajjad's.

#### `@next/third-parties` (^14.1.3)
- **Purpose:** Optimized third-party script loading (Google Tag Manager).
- **Used:** Yes — `layout.js` renders `<GoogleTagManager gtmId={...} />`.
- **Should remove:** No — keep if analytics are needed.
- **Should upgrade:** Align version with Next.js 14.x or upgrade entire stack to Next 15 together.

#### `axios` (^1.6.8)
- **Purpose:** HTTP requests in API routes (Telegram, reCAPTCHA verification).
- **Used:** Partially — `app/api/contact/route.js`, `app/api/google/route.js` only.
- **Should remove:** Could replace with native `fetch` in API routes to reduce bundle (server-only).
- **Should upgrade:** Yes — 1.6.8 is dated; current 1.x has security patches.

#### `lottie-react` (^2.4.0)
- **Purpose:** Render Lottie JSON animations in Experience and Education sections.
- **Used:** Yes — via dynamic import in `animation-lottie.jsx`.
- **Should remove:** No — valuable for personality; consider Motion for UI micro-interactions separately.
- **Should upgrade:** Check for React 19 compatibility when upgrading Next.

#### `next` (^14.2.15)
- **Purpose:** Core framework.
- **Used:** Yes.
- **Should remove:** No.
- **Should upgrade:** **Yes — to Next.js 15** as part of Phase 1 modernization (App Router maturity, caching APIs, React 19).

#### `nodemailer` (^6.9.15)
- **Purpose:** Send email from `/api/contact` route via Gmail SMTP.
- **Used:** Code exists in API route — **frontend never calls this route**.
- **Should remove:** Not until contact form is migrated; otherwise yes (dead weight).
- **Should upgrade:** Consider Resend, Postmark, or SendGrid for production reliability.

#### `react` / `react-dom` ("latest" → 18.2.0)
- **Purpose:** UI rendering.
- **Used:** Yes.
- **Should remove:** No.
- **Should upgrade:** Pin explicit versions; upgrade to React 19 with Next 15.

#### `react-fast-marquee` (^1.6.2)
- **Purpose:** Infinite horizontal scroll for skills cards.
- **Used:** Yes — `skills/index.jsx`.
- **Should remove:** Could replace with CSS animation or Motion-powered marquee for better control.
- **Should upgrade:** Low priority.

#### `react-google-recaptcha` (^3.1.0)
- **Purpose:** Bot protection on contact form.
- **Used:** **No** — API route `/api/google` exists but no UI integration.
- **Should remove:** **Yes** — unless implementing CAPTCHA in redesign.
- **Should upgrade:** N/A if removed; use `@google-cloud/recaptcha-enterprise` or hCaptcha/Turnstile for modern stack.

#### `react-icons` (^4.11.0)

- **Used:** Yes — extensively.
- **Should remove:** Could consolidate to `lucide-react` for consistency and tree-shaking.

#### `react-toastify` (^10.0.6)

- **Used:** Yes — `layout.js` + `contact-form.jsx`.
- **Should remove:** Could replace with **Sonner** (lighter, better aesthetics).
- **Should upgrade:** Or replace during redesign.

#### `sharp` (^0.33.5)

## Appendix F: Deep-Dive Section Audits

### Hero Section — Pixel-Level Breakdown

The hero occupies `app/components/homepage/hero-section/index.jsx` (~200 lines). It is a **Server Component** with no interactivity beyond link navigation.

**Layout structure:**
- Outer `<section>` with `py-4 lg:py-12` — relatively compact vertical padding for a hero; premium portfolios typically use `min-h-screen` or `min-h-[90vh]` for dramatic first viewport.
- Background: `hero.svg` positioned `absolute -top-[98px] -z-10` — creates decorative grid/wave behind content but is not responsive to viewport height; may clip oddly on ultra-wide monitors.
- Grid: `grid-cols-1 lg:grid-cols-2` — single column mobile, two columns desktop. Order swap (`order-1` / `order-2`) puts code window **above** text on mobile — prioritizes gimmick over value proposition for mobile recruiters.

**Typography analysis:**
- H1 uses three color spans: white base, `pink-500` for name, `#16f2b3` for designation.
- Copy reads: "This is **Sajjad Karim**, I'm a Professional **Mern Stack Developer**." — grammatically awkward and "Mern" should be "MERN" for industry standard.
- No subheadline, no location, no availability status, no years of experience counter.

**Social proof row:**
- Five icon slots but only four rendered (Leetcode imported, never rendered; `personalData.leetcode` is empty string).
- All icons identical treatment: pink, scale on hover — no platform-specific brand colors, no tooltips, no follower counts.

**CTA analysis:**
- "Contact me" uses outline gradient border technique — visually secondary.
- "Get Resume" uses filled gradient — visually primary. For job-seeking, this priority may be correct, but both CTAs are similar size.
- Contact links to `#contact` anchor — good.
- Resume links to `personalData.resume` (`./resume.pdf`) — relative path may break from `/blog` routes; should be `/resume.pdf`.

**Code window panel:**
- macOS-style traffic light dots — familiar developer aesthetic.
- Hardcoded skills array in JSX — **does not sync** with `skillsData`.
- `hireable()` function — overused portfolio trope since ~2018.
- `blink` class — **CSS never defined** — incomplete feature.

## Appendix S: Tablet & Responsive Breakpoint Analysis

### Tailwind Breakpoints in Use

| Breakpoint | Pixels | Usage in Portfolio |
|------------|--------|-------------------|
| `default` | 0px+ | Mobile-first base styles |
| `sm:` | 640px | Padding, grid columns (blog), text sizes |
| `md:` | 768px | **Navbar visibility toggle**, text sizes, padding |
| `lg:` | 1024px | Two-column layouts, decorative side labels, py spacing |
| `xl:` | 1280px | Container max-width stepping |
| `2xl:` | 1536px | Container max-width `92rem` |
| `4k:` | 1980px | **Defined in config but never used in any component** |

### Tablet-Specific Issues (768px–1023px)

1. **Navbar appears at 768px** but iPad portrait (768px) may still feel cramped with 6 uppercase links + logo.
2. **Two-column layouts wait until `lg:` (1024px)** — tablet users see single-column for about, experience, education, contact. Wasted horizontal space.
3. **Rotated section labels** (`ABOUT ME`, `CONTACT`) hidden until `lg:` — tablet loses wayfinding visual cues.
4. **Skills marquee** full-width on tablet — acceptable but cards may feel small at `w-36`.
5. **Sticky project cards** on tablet — stacking effect works but card max-width `max-w-2xl` leaves margins.
6. **Hero code window** appears above text between 768px–1023px if order swap applies at lg only — actually order swap is at lg, so tablet still has text first. Good.

### Recommended Breakpoint Strategy for Redesign

- Introduce `md:` two-column for about/contact at 768px.
- Add `sm:` hamburger menu instead of waiting for full horizontal nav.
- Use container queries on project cards for component-level responsiveness.
- Implement `4k:` max-width cap with increased whitespace for ultra-wide monitors.
